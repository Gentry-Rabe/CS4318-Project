# Project Phase 2 — mC Parser & AST

**CS 4318 — Compiler Construction**

## Overview

Build a parser for the **mC** language (a C-like language) that:

1. Consumes the token stream from the Phase 1 lexical analyzer (`scanner.l`)
2. Detects and reports **syntax errors** with line numbers
3. Constructs an **Abstract Syntax Tree (AST)** for valid programs
4. Builds a **symbol table** tracking each identifier's name, type, and scope

## Repository Structure

```
├── src/
│   ├── scanner.l      # Lexer (reuse from Phase 1)
│   ├── parser.y       # YACC grammar — implement rules and AST construction here
│   ├── tree.c / .h    # AST node definitions and printAst() — implement here
│   ├── strtab.c / .h  # Symbol table (djb2 hash, ST_insert, ST_lookup) — implement here
│   └── driver.c       # Entry point, calls printAst() — provided, do not modify
├── test/
│   ├── cases/         # Sample .mC input programs
│   └── exp/           # Expected output for each test case
└── Makefile
```

> **Note:** Token definitions previously in `tokendef.h` are now generated by `parser.y`. Do not use a separate `tokendef.h`.

## Building

```sh
make clean && make
```

The binary is written to `obj/mcc`.

## Running

The compiler accepts input in four ways:

```sh
./obj/mcc input.mC              # file argument
./obj/mcc < input.mC            # file redirect
cat input.mC | ./obj/mcc        # pipe
./obj/mcc                       # interactive (Ctrl-D to end)
```

## Testing

```sh
make test
```

This runs all cases in `test/cases/` against the expected output in `test/exp/`.

## What to Implement

### `parser.y`
- Express the mC grammar as YACC rules
- Resolve any shift/reduce or reduce/reduce conflicts without changing grammar semantics
- Call `ST_insert` on every declaration (`varDecl`, `formalDecl`, `funDecl`)
- Call `ST_lookup` on every use (`var`, `funCallExpr`) and emit an undeclared-symbol warning if not found
- Build and connect AST nodes using `maketree` / `maketreeWithVal` / `addChild`

### `tree.c`
- Implement `maketree(kind)` — allocates a node with zero children
- Implement `maketreeWithVal(kind, val)` — allocates a leaf node storing a value
- Implement `addChild(parent, child)` — wires a child into the parent node
- Implement `printAst(root, nestLevel)` — recursively prints the AST, indented by 2 spaces per level; include extra info for identifiers, integer/char/string constants, operators, and type specifiers (see expected output files for exact format)

### `strtab.c`
- Implement `ST_insert(id, scope, data_type, symbol_type)` — hash key = djb2(`scope` + `id`) % `MAXIDS`
- Implement `ST_lookup(id, scope)` — search by the same key; return index or -1 if not found
- Symbol table entry fields: `id`, `scope` (function name or blank for globals), `data_type` (`INT_TYPE`, `CHAR_TYPE`, `VOID_TYPE`), `symbol_type` (`SCALAR`, `ARRAY`, `FUNCTION`)

## AST Output Format

Each node prints on its own line. Children are indented +2 spaces relative to their parent. Nodes carrying extra information use a comma-separated suffix:

```
program
  declList
    decl
      funDecl
        funcTypeName
          typeSpecifier,void
          identifier,main
        funBody
          ...
            addop,+
            integer,3
```

The symbol table prints after the AST:

```
SYMBOL TABLE:
91: int main:i
242: void :main()
```

(Hash indices will vary with your implementation — this is expected per the assignment.)

## Error Handling

| Error | Behavior |
|---|---|
| Syntax error | Print line number where the program deviates from the grammar |
| Undeclared symbol | Print a warning (not a hard error at this phase) |
| Scanner errors (unterminated comment/string, bad escape, leading-zero int, id starting with digit) | Report line + column number |
